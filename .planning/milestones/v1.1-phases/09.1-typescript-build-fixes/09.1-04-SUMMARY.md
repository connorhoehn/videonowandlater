## Decisions Made
- Used named constructor functions with `this`-assignment (not `return { ... }`) for mock factories in replenish-pool so that `constructor.name` equals the function name (e.g. `IvsClient`) rather than `Object`. When a constructor function returns a plain object via `return`, the `new` expression ignores the returned value and the prototype chain is lost.
- Mocked session-repository and resource-pool-repository in recording-ended test (not just dynamodb-client) because the handler calls repository functions directly, bypassing the client mock.
- Added `RECORDING_CONFIGURATION_ARN` to replenish-pool handler beforeEach — the handler reads this env var and throws if it is missing.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Constructor factory pattern used `return {}` instead of `this` assignment**
- **Found during:** Task 3 (replenish-pool mock implementation)
- **Issue:** Plan's mock factory used `function IvsClient() { return { send: ... } }` — when called with `new`, a constructor function that returns a plain object causes `new` to discard the returned value and produce an instance of the function itself, but with no `this` properties. The resulting object has `constructor.name === 'Object'`, not `'IvsClient'`.
- **Fix:** Changed all constructor functions to use `this.send = jest.fn()...` so the `new` expression creates an instance with the correct prototype chain and `constructor.name` matches the function name.
- **Files modified:** backend/src/handlers/__tests__/replenish-pool.test.ts
- **Verification:** All 6 replenish-pool tests pass including 4 singleton constructor.name assertions
- **Committed in:** fea97aa (Task 3 commit)
