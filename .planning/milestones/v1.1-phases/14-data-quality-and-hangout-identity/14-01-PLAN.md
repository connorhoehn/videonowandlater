# Plan 14-01: Data Quality Filter and Hangout Identity Fix

## Objective
Fix two data quality gaps from the v1.1 audit:
1. REPLAY-01: Home feed shows only `recordingStatus='available'` recordings (no more "Awaiting recording..." stubs)
2. HANG-13: HangoutPage passes `cognito:username` as `sessionOwnerId` to ChatPanel (not the `sub` UUID)

## Root Cause Analysis

### REPLAY-01
`getRecentRecordings` in `session-repository.ts` scans with `FilterExpression: '#status = :ended AND begins_with(PK, :pk)'`. Sessions with `status='ended'` but `recordingStatus='processing'|'failed'|'pending'` are included in results. These show as permanently non-interactive "Awaiting recording..." tiles on the home feed.

Fix: Add `AND recordingStatus = :available` to the FilterExpression.

### HANG-13
`create-session.ts` and `create-chat-token.ts` both use `cognito:username` as the user identity. So `session.userId` = `cognito:username` and IVS Chat `message.sender.userId` = `cognito:username`.

`HangoutPage.tsx` extracts `idToken.payload.sub` (UUID) as `userId` state and passes it as `sessionOwnerId` to ChatPanel. `MessageList` checks `message.sender.userId === sessionOwnerId` to badge broadcaster messages â€” this comparison is always false because `cognito:username !== sub`.

Fix: Extract `idToken.payload['cognito:username']` instead of `sub` for the `userId` state in HangoutPage.
