# Phase 22.1: API Gateway Wiring for Playback Token Generation

## Problem

Phase 22 implementation is **95% complete** but has a **critical gap**: the `generate-playback-token` handler exists, is tested (8/8 passing), and ready for deployment, but it is **not exposed via API Gateway**.

**Current state:**
- Handler implemented: `backend/src/handlers/generate-playback-token.ts` (148 lines, tested)
- Infrastructure environment variables set: `IVS_PLAYBACK_PRIVATE_KEY` read in api-stack.ts (lines 467-477) but not passed to any Lambda
- API endpoint missing: `POST /sessions/{sessionId}/playback-token` does not exist in CDK stack

**Impact:**
- Broadcasters cannot call the endpoint to generate tokens
- Phase 22 verification truth 3 fails: "Broadcaster can generate shareable viewing links (tokens) to send to specific viewers"
- All backend logic complete, but API surface unreachable

**Root cause:**
Plan 22-02 implemented the handler, Plan 22-03 prepared environment infrastructure, but neither added the CDK wiring to expose the endpoint. This is a pure infrastructure/wiring task, not implementation.

## Solution

Add **25-30 lines** to `infra/lib/stacks/api-stack.ts`:

1. Create Lambda function definition using `NodejsFunction` for generate-playback-token handler (lines 225-237, after create-chat-token)
2. Pass environment variables: `TABLE_NAME` and `IVS_PLAYBACK_PRIVATE_KEY`
3. Grant DynamoDB read permissions (handler only reads session metadata)
4. Create API Gateway resource: `/sessions/{sessionId}/playback-token`
5. Add POST method with Cognito authorizer (authenticated endpoint, broadcasters only)

**Pattern reference:** Use `create-chat-token` handler wiring (lines 200-223) as the exact CDK pattern to follow.
