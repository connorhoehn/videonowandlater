## Key Context

**From STATE.md — Schema Pitfalls:**
- `start-transcribe` swallows transient errors (line 87-90) — currently catches all errors without distinguishing transient from permanent
- `recording-ended` receives events from three sources: IVS broadcast, IVS RealTime stage, and recovery events with different shapes
- Permanent schema failures must NOT be rethrown (prevents infinite retry loops)
- Transient API errors (ThrottlingException) must be rethrown to allow SQS retry

**From RESEARCH.md — Standard Patterns:**
- Use Zod `z.discriminatedUnion` for multi-source events (recording-ended with broadcast/hangout/recovery shapes)
- Use `z.safeParse()` at SQS record boundary, NOT inside processEvent
- Add failed messageIds to `batchItemFailures` array to route to DLQ
- Distinguish transient (ThrottlingException) from permanent (missing field) errors in start-transcribe

  <action>
    Create one schema file per handler with the following contracts. Use Zod patterns from RESEARCH.md:

    **recording-ended.schema.ts** — Use `z.discriminatedUnion` to handle broadcast, hangout, and recovery event shapes:
    - Broadcast shape: `{ channel_name: string, stream_id: string, recording_status: enum, recording_s3_bucket_name: string, recording_s3_key_prefix: string, recording_duration_ms: number }`
    - Hangout shape: `{ event_name: 'Recording End', session_id: string, participant_id: string, recording_s3_bucket_name: string, recording_s3_key_prefix: string, recording_duration_ms: number }`
    - Recovery shape: `{ recoveryAttempt: true, sessionId: string, recoveryAttemptCount: number }`
    - Use discriminator field 'event_name' for union detection
    - Export type `RecordingEndedDetail` for TypeScript inference

    **start-transcribe.schema.ts** — Simple object schema:
    - `{ sessionId: string (required), recordingHlsUrl: string (required) }`
    - Validate URLs end with .m3u8
    - Export type `UploadRecordingAvailableDetail`

    **transcode-completed.schema.ts** — MediaConvert job detail:
    - `{ jobName: string (required), status: enum ['SUBMITTED', 'PROGRESSING', 'COMPLETE', 'CANCELED', 'ERROR'], userMetadata: { sessionId: string } }`
    - sessionId must be extracted from userMetadata during validation
    - Export type `MediaConvertJobDetail`

    **transcribe-completed.schema.ts** — AWS Transcribe job detail:
    - `{ TranscriptionJob: { TranscriptionJobStatus: enum ['QUEUED', 'IN_PROGRESS', 'FAILED', 'COMPLETED'], TranscriptionJobName: string, Results: object (any) } }`
    - Export type `TranscribeJobDetail`

    **store-summary.schema.ts** — Summary store trigger:
    - `{ sessionId: string (required), transcriptS3Uri: string (required) }`
    - Export type `TranscriptStoreDetail`

    **on-mediaconvert-complete.schema.ts** — MediaConvert completion event:
    - `{ jobName: string (required), status: enum }`
    - Export type `MediaConvertCompleteDetail`

    Update `backend/src/handlers/schemas/index.ts` to re-export all schemas and types for barrel import.

    **Pattern guidance from RESEARCH.md:**
    - Use `z.object({ ... })` for each schema
    - Use `z.enum(['VALUE1', 'VALUE2'])` for enum fields
    - Use `z.number().positive()` for duration fields
    - Use `z.string().url()` for URLs (or custom validation for S3 paths)
    - Use `z.discriminatedUnion('field', [...])` for recording-ended
    - Export both schema and `z.infer<typeof schema>` type
  </action>
