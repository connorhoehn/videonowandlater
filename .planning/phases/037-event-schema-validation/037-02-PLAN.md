Implement Zod schema validation at the SQS boundary in recording-ended and transcode-completed handlers. These handlers manage complex event shapes (broadcast/hangout/recovery for recording-ended, MediaConvert job details for transcode-completed) and are the highest-risk entry points.

**Purpose:** Prevent malformed events from causing side effects (DynamoDB writes, S3 operations) by validating at the handler boundary before processEvent is called.

- Validation failures logged with specific field names and received values
- Invalid records added to batchItemFailures (routed to DLQ)
- Handler receives fully typed events (no `as any` casts)

## Key Context

**recording-ended complexity:**
- Receives events from 3 sources: IVS broadcast, IVS RealTime stage, and recovery cron
- Each source has a different event shape
- Use `z.discriminatedUnion('event_name', [...])` from schema definition

**transcode-completed complexity:**
- MediaConvert job details nested in eventBridge detail field
- sessionId is in `userMetadata.sessionId` (not at top level)
- Must validate job status enum and extract sessionId

**Validation pattern (from RESEARCH.md):**
```
1. Parse JSON from record.body
2. Validate EventBridge envelope (has `detail` field)
3. Validate custom detail schema
4. On validation failure: log with fieldErrors, add record.messageId to failures[], continue to next record
5. On validation success: call processEvent with typed detail object
```

**From STATE.md:**
- recording-ended is 20K lines (large handler, many edge cases) — focus validation on boundary only
- Both handlers are traced (x-ray active), so validation failures will be visible in traces

**Logging pattern:**
```javascript
const fieldErrors = result.error.flatten().fieldErrors;
logger.error('Event validation failed', {
  messageId: record.messageId,
  validationErrors: Object.entries(fieldErrors).map(([field, messages]) => ({
    field,
    issues: messages,
  })),
});
```

