    ```typescript
    try {
      const session = await getSessionById(tableName, sessionId);
      if (session?.transcriptStatus === 'available' && session?.transcript) {
        logger.info('Transcript already available (idempotent retry)', { sessionId });
        // No-op: SQS message acknowledged below (batchItemFailures empty)
        return;
      }
    } catch (error: any) {
      logger.warn('Failed to check session state (non-blocking, continue):', { errorMessage: error.message });
      // If we can't verify, proceed with normal flow — better to re-write than to silently skip
    }
    ```

    This guard:
    - Checks if `session.transcriptStatus === 'available'` AND `session.transcript` is present (both required to confirm prior execution)
    - If both true: logs "idempotent retry", returns early (no further processing)
    - If false: proceeds to normal flow (first execution or recovery from previous failure)
    - If getSessionById fails: logs warning but continues (fail-open: better to re-process than lose data)

    Important: The early `return` in handler means SQS message is acknowledged (batchItemFailures remains empty) and handler exits without throwing.

    Why this approach:
    - Session state is the authoritative truth (single source)
    - Avoids separate idempotency table lookup (reduces latency)
    - Already fetching session anyway in normal flow for other fields
