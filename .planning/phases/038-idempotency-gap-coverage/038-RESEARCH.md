# Phase 38: Idempotency Gap Coverage - Research

**Researched:** 2026-03-13
**Domain:** AWS Lambda idempotency patterns, DynamoDB persistence, SQS at-least-once delivery
**Confidence:** HIGH

## Summary

Phase 38 closes the idempotency gap in two pipeline handlers: `transcribe-completed` and `store-summary`. These handlers currently lack guards against SQS duplicate delivery and concurrent invocation, exposing them to re-execution of expensive operations (S3 writes, Bedrock invocations). The solution uses AWS Lambda Powertools Idempotency utility with DynamoDB persistence, following the pattern already established in `transcode-completed` but with a critical difference: the two target handlers perform side effects OUTSIDE Powertools' automatic wrapper (S3/Bedrock calls in `store-summary`), requiring custom idempotency guards based on session state rather than function-level decorators.

**Primary recommendation:** Implement idempotency guards by checking session state BEFORE side effects (e.g., check if transcript already exists in S3 before calling `updateTranscriptStatus`; check if summary already exists before invoking Bedrock). Use stable business identifiers (`sessionId`) as idempotency keys, NOT SQS `messageId` (which changes on DLQ re-drive). Store idempotency records in a dedicated `vnl-idempotency` DynamoDB table with TTL, using Powertools persistence layer for standardization.

## Phase Requirements

| ID | Description | Research Support |
|----|-------------|-----------------|
| IDEM-01 | Re-driving a transcribe-completed message that already ran (transcript stored in S3) produces no second S3 write and no error | Session state checking: verify `transcriptStatus === 'available'` and transcript file exists before S3 write |

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Idempotency table structure:**
- Use `vnl-idempotency` DynamoDB table (exact name — Powertools writes to this)
- Required attributes: partition key `id` (auto-generated by Powertools), `expiration` (TTL, exact name for Powertools auto-cleanup)
- Idempotency key MUST be stable business identifier (`sessionId`), NOT SQS `messageId` — messageId changes on DLQ re-drive, bypassing entire guard

**Registration requirement:**
- `registerLambdaContext` is mandatory in handler invocation — without it, INPROGRESS records block retries after Lambda timeouts, preventing recovery
- Powertools requires `registerLambdaContext(context)` called in every invocation to properly track timeout boundaries

**Existing backstop pattern:**
- Keep the existing `ConflictException` catch in `transcode-completed` as a belt-and-suspenders approach — Powertools provides one layer, Transcribe job name uniqueness provides second layer

- DynamoDB table provisioning (on-demand vs. provisioned capacity) — no requirement, but on-demand recommended for variable pipeline load
- Response caching strategy (in-memory vs. DynamoDB-only) — Powertools defaults suitable

### Deferred Ideas

- Full Powertools migration of all 5 handlers (Phase 38 scope: 2 handlers only)

### Pattern 1: Idempotency Guard via Session State Check (Recommended for Phase 38)
**What:** Before expensive operation, check session state to determine if operation already completed. If completed, return success (no-op). If not completed, execute operation.

**When to use:** When the data model provides authoritative completion state that's faster to check than a separate idempotency table lookup (e.g., `transcriptStatus === 'available'`).

**Example:**
```typescript
// From Phase 37 schema validation
const { sessionId, transcriptS3Uri } = event.detail;
const tableName = process.env.TABLE_NAME!;

try {
  // Check 1: Session already has transcript available?
  const session = await getSessionById(tableName, sessionId);
  if (session?.transcriptStatus === 'available' && session?.transcript) {
    // Idempotent no-op: transcript already stored in previous execution
    logger.info('Transcript already available (idempotent skip)', { sessionId });
    return; // Success — SQS message acknowledged
  }

  // Side effect: Update session with transcript (first execution only)
  await updateTranscriptStatus(tableName, sessionId, 'available', s3Uri, plainText);
  logger.info('Transcript stored', { sessionId });
} catch (error: any) {
  // Permanent vs. transient error handling (unchanged from Phase 37)
  logger.error('Failed to store transcript:', { errorMessage: error.message });
  throw error;
}
```

**Tradeoff:** One DynamoDB GetItem per invocation (vs. zero for idempotent hits with Powertools). Acceptable because session lookups are required anyway to populate other fields. Eliminates need for separate idempotency table lookup.

### Pattern 2: Powertools Decorator (For Reference — Not Recommended for Phase 38 SQS Handlers)
**What:** Wrap entire handler with `@idempotent` decorator; Powertools automatically deduplicates based on event payload hash.

**When NOT to use for Phase 38:** SQS handlers forward an EventBridge event nested in the body. Event `id` field changes on each SQS retry, causing hash collision detection to fail. Powertools expects stable event ID, which SQS doesn't guarantee.

**Example (reference from transcode-completed):**
```typescript
@idempotent({ persistenceStore })
async function processEvent(event: EventBridgeEvent<string, TranscribeJobDetail>): Promise<void> {
  // Body of handler — idempotency managed automatically
}
```

**Why it fails for transcribe-completed/store-summary:**
- SQS MessageId changes on DLQ re-drive
- EventBridge event `detail.sessionId` is stable, but nested in body
- Powertools hashes the entire SQS body, which includes MessageId — hash changes even for same logical message

### Pattern 3: Concurrent Delivery Protection (IDEM-03)
**What:** Detect when two Lambda instances process the same message simultaneously and allow only one to complete.

**When to use:** When same message delivered to handler twice with <1s interval (race condition).

**How Powertools handles it:**
1. First invocation: PutItem with `status='INPROGRESS'`, then UpdateItem to `status='COMPLETE'` after side effect succeeds
2. Second invocation (concurrent): GetItem returns `status='INPROGRESS'`, waits up to `maxRetries` * `waitSeconds` (configurable, default 15s), then throws `IdempotencyAlreadyInProgressError`
3. Handler catches error, returns success (idempotent no-op)

**In Phase 38 context:** Session state check automatically prevents concurrent execution — two handlers checking `session.transcriptStatus === 'available'` will both see `'processing'` initially, then first to complete updates to `'available'`, second reads `'available'` on retry.

### Anti-Patterns to Avoid

- **Using SQS `messageId` as idempotency key:** MessageId changes on DLQ re-drive, defeating idempotency entirely. Always extract stable business ID from event.detail (sessionId in our case).

- **Storing idempotency records in session table:** Mixing concerns. Separate idempotency table has different access patterns (high write-on-first-attempt, low read-on-retry) and TTL policies (idempotency records expire after 1h, session records never auto-expire).

- **Skipping `registerLambdaContext(context)` in handler:** Without it, if handler times out during side effect, Powertools doesn't know Lambda killed the function. On retry, it keeps returning cached response from first partial execution, losing durability guarantee.

- **No-op returning error instead of success:** If duplicate execution succeeds idempotently (no-op), return `{ batchItemFailures: [] }` to acknowledge message. Throwing error causes SQS to retry, defeating idempotency.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tracking which operations completed | Custom DynamoDB "seen" table with manual atomic updates | Powertools Idempotency (handles INPROGRESS → COMPLETE atomicity) | Powertools handles race conditions between Lambda timeout + concurrent retry; custom approach requires ConditionalExpression, RetryAnalytics, TTL policies |
| Detecting concurrent duplicate messages | Manual timestamp comparison + sleep loops | Powertools `IdempotencyAlreadyInProgressError` + config.useLocalCache | Local in-memory cache (512 items default) prevents thundering herd on high-concurrency scenarios; Powertools adds jitter to wait times |
| Extracting idempotency key from nested event | Manual JSON path parsing | Zod schema + stable `detail.sessionId` field | Zod validation (Phase 37) ensures sessionId exists before idempotency code runs; manual parsing bypasses that guarantee, crashes on malformed events |
| Implementing timeout-aware idempotency | Custom Lambda context inspection | Powertools `registerLambdaContext(context)` | Powertools tracks remaining execution time; custom approach must re-implement Lambda context duration calculation, remainder, request ID tracking |

**Key insight:** SQS at-least-once delivery is fundamentally non-idempotent without persistent state tracking. Powertools abstracts the complex state machine (INPROGRESS → COMPLETE, timeout recovery, concurrent detection) so handlers only need to define what "already executed" means.

## Common Pitfalls

### Pitfall 1: Idempotency Key Changes on Retry
**What goes wrong:** Using `record.messageId` from SQS as idempotency key. On first attempt, message processed and marked idempotent with key `msg-123`. SQS retries; same message gets `msg-456` (new receipt in DLQ). Handler sees new key, executes duplicate side effect.

**Why it happens:** SQS generates new MessageId on each delivery attempt. DLQ re-drive also generates new MessageId. Only the SQS message body (and `event.detail.sessionId` within it) remains stable.

**How to avoid:** Extract idempotency key from `event.detail` (the EventBridge event nested in SQS body), never from SQS record metadata. `sessionId` is stable across all retries.

**Warning signs:**
- Idempotency table showing duplicate entries with different keys for same session
- Duplicate S3 writes or Bedrock invocations appearing in CloudWatch logs minutes apart
- Test failure when re-driving message from DLQ simulator

### Pitfall 2: INPROGRESS Records Block Recovery After Lambda Timeout
**What goes wrong:** Handler starts Bedrock invocation, Lambda times out at 30s, function killed mid-request. Powertools has recorded `status='INPROGRESS'`. SQS retries; handler checks idempotency, sees INPROGRESS, waits up to 15s for completion, timeout expires, handler throws error, message goes to DLQ. User must manually delete the INPROGRESS record to retry.

**Why it happens:** Powertools relies on `registerLambdaContext(context)` to know when Lambda is about to kill the function. Without it, Powertools doesn't transition INPROGRESS → COMPLETE when timeout occurs.

**How to avoid:** Call `config.registerLambdaContext(context)` at the start of every handler invocation. Powertools uses `context.getRemainingTimeInMillis()` to calculate timeout boundaries and auto-expire INPROGRESS records.

**Warning signs:**
- Idempotency table showing old INPROGRESS records (hours old)
- "IdempotencyAlreadyInProgressError" in logs even after manual retry
- Can only recover by manual DynamoDB delete

### Pitfall 3: Concurrent Delivery Race Condition (IDEM-03 Verification)
**What goes wrong:** Same message delivered to Lambda twice within 100ms (network race). Both invocations call Bedrock before either completes. Handler receives two summaries, stores second one (overwriting first). User sees inconsistent summary depending on which invocation's response wins.

**Why it happens:** SQS can deliver message to multiple consumer Lambda instances if scaling/retries overlap. Powertools INPROGRESS locking prevents most cases, but if first handler crashes after PutItem (before UpdateItem), second invocation still passes lock check.

**How to avoid:**
- Use Powertools with `useLocalCache: true` (default) to prevent same-instance concurrent execution
- Call `registerLambdaContext` so Powertools knows timeout window and doesn't hold locks forever
- Test concurrent delivery scenario in integration tests (see Test Requirements section)

**Warning signs:**
- Multiple CloudWatch log entries from same sessionId within seconds
- S3 timestamp shows two Bedrock invocations milliseconds apart
- Test failure when handler called twice in Promise.all()

### Pitfall 4: Returning Error on Idempotent No-Op
**What goes wrong:** Handler detects duplicate execution, logs "already processed", but throws an error because it "didn't do anything". SQS sees `batchItemFailures: [record.messageId]`, retries message forever.

**Why it happens:** Misunderstanding that idempotent no-op is success, not failure. "Already processed" is the happy path for a retry.

**How to avoid:** Return success (empty `batchItemFailures`) when idempotency guard activates. Throw only on transient/permanent failures (Bedrock unavailable, invalid sessionId).

**Warning signs:**
- SQS metrics show messages with ApproximateReceiveCount > 3
- DLQ message timestamps showing same message re-delivered >10 times
- CloudWatch logs showing same sessionId processed repeatedly

### Pitfall 5: Mixing Idempotency Table and Session Table Cleanup Policies
**What goes wrong:** Store idempotency records in session table. Set session TTL to 90 days. Idempotency records expire after 90 days, but user can still replay 89-day-old messages, causing duplicate Bedrock invocations and unexpected costs.

**Why it happens:** Different data has different lifecycle. Session records represent user activity (live 90+ days). Idempotency records track de-duplication window (live 1 hour, rarely accessed after 5 minutes).

**How to avoid:** Use separate `vnl-idempotency` table with `expiryAttr: 'expiration'` and TTL set to 3600 seconds (1 hour). Session table keeps its own TTL policy independent.

**Warning signs:**
- Idempotency table size growing unbounded (should stay <50MB with 1h TTL)
- Replay tests from day 30 succeeding (no idempotency guard, duplicate execution)
- Cost spike if accidental Bedrock re-invocation happens

### Example 1: Idempotency Guard in transcribe-completed (Custom Session State Check)

```typescript
// Source: Phase 37 validated event + Phase 38 session state check pattern
// File: backend/src/handlers/transcribe-completed.ts

async function processEvent(
  event: EventBridgeEvent<string, TranscribeJobDetail>,
  tracer: Tracer
): Promise<void> {
  const { sessionId } = event.detail; // sessionId already extracted via Zod (Phase 37)
  const tableName = process.env.TABLE_NAME!;
  const transcriptionBucket = process.env.TRANSCRIPTION_BUCKET!;

  tracer.putAnnotation('sessionId', sessionId);
  logger.appendPersistentKeys({ sessionId });
  logger.info('Pipeline stage entered', { jobName, transcriptionJobStatus: detail.TranscriptionJobStatus });

  // IDEM-01: Check if transcript already available (idempotent guard)
  try {
    const session = await getSessionById(tableName, sessionId);
    if (session?.transcriptStatus === 'available' && session?.transcript) {
      logger.info('Transcript already available (idempotent retry)', { sessionId });
      // No-op: SQS message acknowledged below (batchItemFailures empty)
      return;
    }
  } catch (error: any) {
    logger.warn('Failed to check session state (non-blocking, continue):', { errorMessage: error.message });
    // If we can't verify, proceed with normal flow — better to re-write than to silently skip
  }

  // Normal flow: fetch transcript, store in S3 and DynamoDB
  if (detail.TranscriptionJobStatus === 'FAILED') {
    try {
      await updateTranscriptStatus(tableName, sessionId, 'failed');
    } catch (error: any) {
      logger.error('Failed to update transcript status:', { errorMessage: error.message });
    }
    return;
  }

  // Fetch transcript from S3 and process...
  const s3Uri = `s3://${transcriptionBucket}/${sessionId}/transcript.json`;
  await updateTranscriptStatus(tableName, sessionId, 'available', s3Uri, plainText);
  logger.info('Transcript stored for session:', { sessionId, s3Uri });
}
```

### Example 2: Idempotency Guard in store-summary (Before Bedrock Invocation)
```typescript
// Source: store-summary handler + Phase 37 schema validation
// File: backend/src/handlers/store-summary.ts

async function processEvent(
  event: EventBridgeEvent<string, TranscriptStoreDetail>,
  tracer: Tracer
): Promise<void> {
  const { sessionId, transcriptS3Uri } = event.detail; // Zod validated (Phase 37)
  const tableName = process.env.TABLE_NAME!;
  const modelId = process.env.BEDROCK_MODEL_ID || 'amazon.nova-lite-v1:0';

  tracer.putAnnotation('sessionId', sessionId);
  tracer.putAnnotation('pipelineStage', 'store-summary');
  logger.appendPersistentKeys({ sessionId });
  logger.info('Pipeline stage entered', { transcriptS3Uri });

  try {
    // IDEM-02: Check if summary already available (idempotent guard)
    const session = await getSessionById(tableName, sessionId);
    if (session?.aiSummaryStatus === 'available' && session?.aiSummary) {
      logger.info('AI summary already available (idempotent retry)', {
        sessionId,
        existingLength: session.aiSummary.length
      });
      // No-op: return success without Bedrock invocation
      return;
    }
  } catch (error: any) {
    logger.warn('Failed to check session state (non-blocking, continue):', { errorMessage: error.message });
    // If we can't verify, proceed — better to re-invoke Bedrock than silently skip with stale data
  }

  // Normal flow: fetch transcript, invoke Bedrock, store summary
  const s3Match = transcriptS3Uri.match(/^s3:\/\/([^/]+)\/(.+)$/);
  if (!s3Match) {
    throw new Error(`Invalid S3 URI format: ${transcriptS3Uri}`);
  }

  const [, bucketName, key] = s3Match;
  const s3Response = await s3Client.send(new GetObjectCommand({
    Bucket: bucketName,
    Key: key,
  }));

  const transcriptText = await s3Response.Body?.transformToString();
  if (!transcriptText || transcriptText.trim().length === 0) {
    logger.warn('Empty transcript from S3:', { sessionId, transcriptS3Uri });
    await updateSessionAiSummary(tableName, sessionId, { aiSummaryStatus: 'failed' });
    return;
  }

  // Prepare Bedrock payload based on model
  const systemPrompt = 'Generate a concise one-paragraph summary (2-3 sentences) of the following video session transcript.';
  const userPrompt = `Transcript:\n\n${transcriptText}`;

  const isClaudeModel = modelId.startsWith('anthropic.');
  const payload = isClaudeModel
    ? { anthropic_version: 'bedrock-2023-05-31', max_tokens: 500, system: systemPrompt, messages: [{ role: 'user', content: [{ type: 'text', text: userPrompt }] }] }
    : { messages: [{ role: 'user', content: [{ text: `${systemPrompt}\n\n${userPrompt}` }] }], inferenceConfig: { maxTokens: 500, temperature: 0.7 } };

  const command = new InvokeModelCommand({
    contentType: 'application/json',
    body: JSON.stringify(payload),
    modelId,
  });

  const apiResponse = await bedrockClient.send(command);
  const decodedResponseBody = new TextDecoder().decode(apiResponse.body);
  const responseBody = JSON.parse(decodedResponseBody);

  const summary = isClaudeModel
    ? responseBody.content[0].text
    : responseBody.output.message.content[0].text;

  // Store summary in DynamoDB
  try {
    await updateSessionAiSummary(tableName, sessionId, {
      aiSummary: summary,
      aiSummaryStatus: 'available',
    });
    logger.info('AI summary stored:', { sessionId, summaryLength: summary.length });
  } catch (storeError: any) {
    logger.error('Failed to store AI summary (non-blocking):', { errorMessage: storeError.message });
    // Non-blocking: summarization succeeded, storage failed — don't throw
  }
}
```

### Example 3: Handler Entry Point with Validation + Idempotency
```typescript
// Source: Phase 37 (validation) + Phase 38 (handler structure)
// File: backend/src/handlers/store-summary.ts (handler function)

export const handler = async (event: SQSEvent): Promise<SQSBatchResponse> => {
  const failures: { itemIdentifier: string }[] = [];

  for (const record of event.Records) {
    const segment = tracer.getSegment();
    const subsegment = segment?.addNewSubsegment(`## ${record.messageId}`) as Subsegment | undefined;
    if (subsegment) tracer.setSegment(subsegment);

    try {
      // Parse JSON from SQS record body
      let ebEvent: any;
      try {
        ebEvent = JSON.parse(record.body);
      } catch (parseError: any) {
        logger.error('Failed to parse SQS record body as JSON', {
          messageId: record.messageId,
          error: parseError.message,
          handler: 'store-summary',
        });
        failures.push({ itemIdentifier: record.messageId });
        continue;
      }

      // Validate EventBridge envelope
      if (!ebEvent.detail) {
        logger.error('EventBridge event missing detail field', {
          messageId: record.messageId,
          handler: 'store-summary',
        });
        failures.push({ itemIdentifier: record.messageId });
        continue;
      }

      // Phase 37: Validate schema
      const parseResult = TranscriptStoreDetailSchema.safeParse(ebEvent.detail);
      if (!parseResult.success) {
        const fieldErrors = parseResult.error.flatten().fieldErrors;
        logger.error('Invalid transcript store detail', {
          messageId: record.messageId,
          handler: 'store-summary',
          fieldErrors,
          detail: JSON.stringify(ebEvent.detail),
        });
        failures.push({ itemIdentifier: record.messageId });
        continue;
      }

      // Phase 38: Process with idempotency guard inline
      const typedEvent: EventBridgeEvent<string, TranscriptStoreDetail> = {
        ...ebEvent,
        detail: parseResult.data,
      };

      // processEvent includes idempotency check — if already processed, returns early
      await processEvent(typedEvent, tracer);
    } catch (err: any) {
      tracer.addErrorAsMetadata(err as Error);
      logger.error('Failed to process SQS record', {
        messageId: record.messageId,
        error: err.message,
      });
      failures.push({ itemIdentifier: record.messageId });
    } finally {
      subsegment?.close();
      if (segment) tracer.setSegment(segment);
    }
  }

  return { batchItemFailures: failures };
};
```

## Open Questions

1. **Should idempotency key be `sessionId` or `sessionId + jobId` (composite)?**
   - What we know: sessionId alone is sufficient if session.transcriptStatus is the source of truth
   - What's unclear: Whether same session can have multiple concurrent transcription jobs (e.g., user uploads video while previous one still transcribing)
   - Recommendation: Use `sessionId` only. If concurrent uploads become a feature, add jobId as composite key in a future phase. Current schema assumes 1 recording per session.

2. **Should IDEM-03 concurrent test use Promise.all or separate Lambda invocations?**
   - What we know: Jest test can invoke handler twice in Promise.all
   - What's unclear: Does session state check prevent concurrent duplicate or just detect it?
   - Recommendation: Use Promise.all for unit test (simulates concurrency); add integration test later in Phase 39 (DLQ tooling) to test actual concurrent Lambda instances.

3. **What if Bedrock invocation partially succeeds (connection timeout during response streaming)?**
   - What we know: store-summary catches all errors and marks aiSummaryStatus='failed'
   - What's unclear: Can Bedrock be partially charged for streaming timeout? Should we log API cost separately?
   - Recommendation: Out of scope for Phase 38 (idempotency focus). Bedrock charges on completion; timeouts don't incur cost. Phase 39 can add detailed error classification.
