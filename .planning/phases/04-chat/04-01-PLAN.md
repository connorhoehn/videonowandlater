# Plan 04-01: Chat Backend API and Message Persistence

## Objective

Implement server-side chat token generation and message persistence with session-relative timestamps, enabling authenticated chat connections and replay synchronization for Phase 5.

## Context

Phase 4 adds real-time text chat to live sessions. This plan delivers the backend foundation: chat token generation (CreateChatToken via AWS SDK), message persistence to DynamoDB with session-relative timestamps, and history retrieval for mid-stream joins. The frontend (Plan 04-02) will consume these endpoints.

**Key decisions from CONTEXT.md:**
- History on join: Last 50 messages
- Server-side token generation only (CHAT-05)
- Session-relative timestamps for replay sync (CHAT-04)

**Dependencies:**
- Resource pool (Phase 2): Sessions already claim ROOM resources with chatRoomArn
- Session lifecycle (Phase 2): Sessions have startedAt timestamp for relative time calculation

<file>backend/src/domain/chat-message.ts</file>
<description>Create ChatMessage domain model</description>
<instructions>
Define ChatMessage interface with fields:
- messageId: string (unique identifier)
- sessionId: string (links to session)
- senderId: string (Cognito username)
- content: string (message text)
- sentAt: string (ISO 8601 timestamp from IVS Chat event)
- sessionRelativeTime: number (milliseconds since session.startedAt - for replay sync)
- senderAttributes: Record<string, string> (displayName, role from token)

Add helper function:
```typescript
export function calculateSessionRelativeTime(sessionStartedAt: string, messageSentAt: string): number {
  const startTime = new Date(sessionStartedAt).getTime();
  const sentTime = new Date(messageSentAt).getTime();
  return sentTime - startTime;
}
```

Use existing imports pattern (no external dependencies needed).
</instructions>
</task>

<file>backend/src/repositories/chat-repository.ts</file>
<description>Create chat repository for message persistence</description>
<instructions>
Implement three functions following existing repository pattern (session-repository.ts):

1. **persistMessage**: Store message in DynamoDB
   - PK: `MESSAGE#{sessionId}`
   - SK: `{sentAt}#{messageId}` (composite for uniqueness, sorted by timestamp)
   - Include all ChatMessage fields
   - Use PutCommand from @aws-sdk/lib-dynamodb

2. **getMessageHistory**: Query last N messages for session
   - KeyConditionExpression: `PK = :pk`
   - ScanIndexForward: false (descending order - newest first)
   - Limit: parameter (default 50)
   - Return reversed array (oldest-first for UI display)
   - Use QueryCommand

3. **getMessageById**: Retrieve single message (for verification/debugging)
   - Use GetCommand with PK and SK

Import getDocumentClient from '../lib/dynamodb-client'.
Follow error handling pattern from existing repositories (console.error + throw).
</instructions>
</task>

<file>backend/src/services/chat-service.ts</file>
<description>Create chat service for token generation business logic</description>
<instructions>
Implement generateChatToken function:

**Signature:**
```typescript
interface GenerateChatTokenRequest {
  sessionId: string;
  userId: string;
  displayName?: string;
}

interface GenerateChatTokenResponse {
  token: string;
  sessionExpirationTime: string;
  tokenExpirationTime: string;
}

export async function generateChatToken(
  tableName: string,
  request: GenerateChatTokenRequest
): Promise<GenerateChatTokenResponse>
```

**Logic:**
1. Fetch session by sessionId using getSessionById from '../repositories/session-repository'
2. Validate session exists (throw error if not found)
3. Extract chatRoomArn from session.claimedResources.chatRoom
4. Determine role: 'broadcaster' if userId === session.userId, else 'viewer'
5. Call CreateChatTokenCommand via getIVSChatClient():
   - roomIdentifier: chatRoomArn
   - userId: request.userId
   - capabilities: ['SEND_MESSAGE', 'DELETE_MESSAGE'] (broadcaster + viewer both can send)
   - sessionDurationInMinutes: 60 (1 hour session)
   - attributes: { displayName: request.displayName || userId, role: role }
6. Return { token, sessionExpirationTime, tokenExpirationTime } from response

Import CreateChatTokenCommand from '@aws-sdk/client-ivschat'.
Import getIVSChatClient from '../lib/ivs-clients'.
Follow service pattern from session-service.ts and broadcast-service.ts.
</instructions>
</task>

<description>Create POST /sessions/{sessionId}/chat/messages handler</description>
<instructions>
Implement APIGatewayProxyHandler for message persistence:

**Route:** POST /sessions/{sessionId}/chat/messages
**Auth:** Cognito authorizer

**Request body:**
```typescript
{
  messageId: string,
  content: string,
  senderId: string,
  senderAttributes: Record<string, string>,
  sentAt: string
}
```

**Logic:**
1. Parse request body (validate JSON)
2. Extract sessionId from pathParameters
3. Fetch session to get startedAt timestamp (for sessionRelativeTime calculation)
4. Validate session exists and status is 'live' (return 400 if not live)
5. Calculate sessionRelativeTime using calculateSessionRelativeTime helper
6. Build ChatMessage object
7. Call persistMessage repository function
8. Return 201 with { messageId, sessionRelativeTime } (confirmation)

Handle errors:
- Invalid JSON: 400
- Session not found: 404
- Session not live: 400 with message "Session must be live to send messages"
- Persistence failure: 500

Follow handler + service pattern from existing handlers.
</instructions>
</task>

## Success Criteria (must_haves)

1. **Server-side token generation (CHAT-05)**: POST /sessions/{sessionId}/chat/token endpoint successfully generates IVS Chat tokens with CreateChatToken API, includes user attributes (displayName, role), and returns token to authenticated frontend clients.

2. **Message persistence with session-relative timestamps (CHAT-04)**: Messages are stored in DynamoDB with PK=MESSAGE#{sessionId}, SK={sentAt}#{messageId}, and sessionRelativeTime field calculated from session.startedAt, enabling replay synchronization in Phase 5.

3. **Chat history retrieval**: GET /sessions/{sessionId}/chat/messages endpoint returns last 50 messages in chronological order (oldest-first) for users joining mid-stream.

4. **API integration**: All three endpoints integrated into API Gateway with Cognito authorization, Lambda handlers deployed, and IAM permissions granted (DynamoDB read/write, ivschat:CreateChatToken).

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| CreateChatToken IAM permission missing | Token generation fails (403) | Add ivschat:CreateChatToken policy in CDK (task 7) |
| Session not yet in 'live' state when message sent | Message persisted with incorrect sessionRelativeTime (null startedAt) | Validate session.status === 'live' in send-message handler (task 5) |
| Message persistence latency | Messages appear in UI but missing from history on refresh | Fire-and-forget pattern acceptable for v1; Phase 5 can add retry logic if needed |
| Chat room ARN mismatch | Token generation fails if session chatRoom field missing | Validate chatRoomArn exists in generateChatToken service (task 3) |

## Notes

- DynamoDB composite sort key pattern ({sentAt}#{messageId}) ensures chronological ordering with uniqueness for concurrent messages
- sessionRelativeTime enables Phase 5 to synchronize chat replay with video scrubbing (e.g., show messages sent at 5:30 timestamp)
- Token sessionDurationInMinutes=60 matches typical broadcast length; Phase 6 may adjust for longer hangouts
- Message persistence via REST endpoint (not DynamoDB Streams) keeps Phase 4 simple; can migrate to Streams pattern if reliability issues arise
