### Functional Summary

**Routing (Task 1):**
- Added `/broadcast/:sessionId` route rendering BroadcastPage wrapped in ProtectedRoute
- Added `/viewer/:sessionId` route rendering ViewerPage wrapped in ProtectedRoute
- Routes are siblings to Layout (not nested) since pages have their own navigation

**Session Creation (Task 2):**
- HomePage now displays blue "Create Broadcast" button above Log Out button
- Button calls `POST /sessions` API with Authorization header
- Loading state shows "Creating..." text with gray background and disabled cursor
- Success navigates to `/broadcast/:sessionId` with session data passed via state
- Error displays inline text "Failed to create session. Try again." with retry on next click
- Error clears automatically when user retries

**API Configuration (Task 3):**
- BroadcastPage and ViewerPage now use `getConfig()?.apiUrl` instead of `import.meta.env.VITE_API_URL`
- Centralized API URL management in aws-config.ts
- Fallback to `http://localhost:3000/api` for local development
- No hardcoded environment variable references remain

## Key Learnings

1. **Pre-existing blocking errors:** The chat components had TypeScript errors from Phase 04-02 that weren't caught because build wasn't run after that phase. Fixed immediately per Deviation Rule 3.

2. **IVS SDK API patterns:** The SendMessageRequest requires instantiation with `new SendMessageRequest(content)`, not a plain object - SDK uses class-based request objects.

3. **Centralized config benefits:** Moving from scattered `import.meta.env.VITE_API_URL` reads to centralized `getConfig()` makes API URL management consistent and easier to change in the future.
