<decisions>
## Implementation Decisions

### Session Creation Flow
- **Loading state**: Button text changes to "Creating..." with spinner while POST /sessions API call is in progress
- **Error handling**: Inline error text appears below button on failure (e.g., "Failed to create session. Try again.")
- **Retry mechanism**: User clicks same "Create Broadcast" button again to retry; error clears on click
- **User input**: Create with backend defaults immediately (no title prompt or configuration modal)
- **API endpoint**: POST /sessions with Authorization header, expects sessionId in response

### Navigation Behavior
- **Post-creation navigation**: Navigate to /broadcast/:sessionId immediately after POST /sessions returns successfully
- **State passing**: Pass session data via `navigate('/broadcast/:id', { state: sessionData })` to avoid redundant fetch on BroadcastPage
- **Browser back button**: Simple back navigation to HomePage (no confirmation, even if camera active)
- **In-app navigation**: No explicit "Home" link on Broadcast/Viewer pages — browser back is sufficient
- **Route structure**: /broadcast/:sessionId for broadcasters, /viewer/:sessionId for viewers

### HomePage Layout & CTA
- **Button text**: "Create Broadcast" (clear, domain-appropriate)
- **Button styling**: Primary action style (blue or green), visually distinct from red "Log Out" button
- **Welcome message**: Keep "Welcome, {username}!" greeting, remove "Future phases will add..." placeholder text
- **Button placement**: Claude's discretion (options: replace placeholder, stack with logout, or separate section)

### Claude's Discretion
- Error handling patterns across the app (chosen in discussion: inline errors, but Claude can implement toast library if it makes sense)
- Exact button placement on HomePage (replace placeholder vs new section)
- Loading spinner implementation (text-based "Creating..." vs icon spinner)
- Exact color values for primary button (any blue/green that contrasts with red logout)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **getConfig() utility** (config/aws-config.ts): Returns AwsConfig with apiUrl, userPoolId, region. Use getConfig().apiUrl for all API calls.
- **ProtectedRoute component** (App.tsx): Already wraps authenticated routes. HomePage is protected, Broadcast/Viewer pages should also use it.
- **useAuth hook** (auth/useAuth.ts): Provides user object and signOut. Already used in HomePage.
- **Layout component** (components/Layout.tsx): Wraps protected routes with shared header/footer. Currently only applied to HomePage route.
- **React Router** (react-router-dom): BrowserRouter, Routes, Route, useNavigate, useParams already in use.

### Established Patterns
- **API calls**: BroadcastPage and ViewerPage currently use `import.meta.env.VITE_API_URL || 'http://localhost:3000/api'`. This pattern needs to be replaced with `getConfig()?.apiUrl` (with null check since getConfig() can return null before config loads).
- **Authentication headers**: localStorage.getItem('token') pattern used in BroadcastPage/ViewerPage. This works but should be consistent across new API calls in HomePage.
- **Inline styles**: HomePage currently uses React inline styles (style prop). Maintain consistency or introduce Tailwind classes (BroadcastPage uses Tailwind, so Tailwind is available).
- **Error boundaries**: No global error handling exists. Inline error state is simplest for this phase.

### Integration Points
- **App.tsx Routes block** (lines 68-80): Add /broadcast/:sessionId and /viewer/:sessionId routes here, inside Layout or as siblings
- **HomePage** (pages/HomePage.tsx): Add "Create Broadcast" button, loading state, error state, and navigation logic
- **BroadcastPage** (features/broadcast/BroadcastPage.tsx line 12): Replace `API_BASE_URL` constant with `getConfig()?.apiUrl`
- **ViewerPage** (features/viewer/ViewerPage.tsx line 11): Replace `API_BASE_URL` constant with `getConfig()?.apiUrl`
- **AuthToken access**: fetchAuthSession() from aws-amplify/auth used in BroadcastPage to get userId. Reuse this pattern if needed for Authorization headers.
