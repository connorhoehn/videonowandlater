## Summary

Phase 4.2 is a frontend integration phase that wires existing components (BroadcastPage, ViewerPage) into App.tsx routing, adds session creation UI to HomePage, and fixes API configuration across components to use centralized `getConfig()` instead of environment variables.

This is primarily a plumbing task using established React patterns. The codebase already has React Router 6 configured, the getConfig() utility exists, and all target components are implemented. The work involves: (1) adding two routes, (2) implementing a button with API call and navigation, (3) replacing hardcoded API_BASE_URL with getConfig().apiUrl.

**Primary recommendation:** Follow existing code patterns (inline styles on HomePage, getConfig() centralized configuration, navigate with state passing) and maintain consistency with established authentication flows.

```tsx
      </ProtectedRoute>

```

**For Phase 4.2:** Add BroadcastPage and ViewerPage as siblings to HomePage route. These could be inside or outside Layout depending on whether they need shared chrome. BroadcastPage and ViewerPage already have their own headers, so they can be siblings outside Layout or inside with Layout rendering Outlet.

### Pattern 2: Protected Route Wrapper
**What:** ProtectedRoute checks authentication before rendering children
**When to use:** Any route that requires login
**Example:**
```tsx
// App.tsx lines 13-31
function ProtectedRoute({ children }: { children: React.ReactNode }) {
  const { isAuthenticated, isLoading } = useAuth();
  if (isLoading) return <div>Loading...</div>;
  return isAuthenticated ? <>{children}</> : <Navigate to="/login" replace />;
}
```

**For Phase 4.2:** Wrap BroadcastPage and ViewerPage routes with ProtectedRoute.

### Pattern 3: Centralized Config with getConfig()
**What:** Load aws-config.json once, access via getConfig() throughout app
**When to use:** Any API call that needs backend URL
**Example:**
```tsx
// aws-config.ts pattern (already exists)
let config: AwsConfig | null = null;

export async function loadConfig(): Promise<AwsConfig | null> {
  const response = await fetch('/aws-config.json');
  config = await response.json();
  return config;
}

export function getConfig(): AwsConfig | null {
  return config;
}

// Usage (replace current pattern):
const apiUrl = getConfig()?.apiUrl;
if (!apiUrl) {
  // Handle config not loaded
  return;
}
fetch(`${apiUrl}/sessions/${sessionId}`)
```

**For Phase 4.2:** Replace `import.meta.env.VITE_API_URL || 'http://localhost:3000/api'` in BroadcastPage.tsx and ViewerPage.tsx with `getConfig()?.apiUrl`.

**What:** Pass data via navigate() to avoid redundant API calls
**When to use:** When creating a resource and immediately navigating to its detail page
**Example:**

**For Phase 4.2:** HomePage creates session, passes data to BroadcastPage via state.

**When to use:** Maintain consistency with existing component patterns

### Anti-Patterns to Avoid
- **Don't hardcode API URLs:** Use getConfig().apiUrl, not import.meta.env.VITE_API_URL
- **Don't refetch on navigation:** Pass data via navigate state when possible
- **Don't mix styling approaches:** HomePage uses inline styles, Broadcast/Viewer use Tailwind; maintain this boundary
- **Don't add confirmation dialogs:** User decision was simple back navigation with no confirmation

## Common Pitfalls

### Pitfall 1: Null Config Access
**What goes wrong:** `getConfig()` returns null before aws-config.json loads, causing API calls to fail with undefined URL
**Why it happens:** App.tsx loads config asynchronously; components may mount before config is ready
**How to avoid:** Always null-check `getConfig()?.apiUrl` before using it
**Warning signs:** TypeError: Cannot read property 'apiUrl' of null

**Example:**
```tsx
// BAD - assumes config is loaded
const apiUrl = getConfig().apiUrl;

// GOOD - handles null case
const config = getConfig();
if (!config?.apiUrl) {
  setError('Configuration not loaded');
  return;
}
const apiUrl = config.apiUrl;
```

### Pitfall 2: Missing Authorization Header
**What goes wrong:** POST /sessions returns 401 Unauthorized
**Why it happens:** Forgetting to include Bearer token in Authorization header
**How to avoid:** Follow existing pattern from BroadcastPage/ViewerPage: `const authToken = localStorage.getItem('token') || '';` then pass in headers
**Warning signs:** 401 errors in network tab

**Example:**
```tsx
// Correct pattern (from ViewerPage.tsx lines 16, 26)
const authToken = localStorage.getItem('token') || '';
fetch(`${apiUrl}/sessions`, {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${authToken}`,
    'Content-Type': 'application/json',
  },
})
```

### Pitfall 3: Route Placement Outside BrowserRouter
**What goes wrong:** Routes don't render, useNavigate/useParams hooks throw errors
**Why it happens:** Adding routes outside <BrowserRouter> context
**How to avoid:** Add routes inside existing <Routes> block (App.tsx lines 68-80)
**Warning signs:** "useNavigate() may be used only in the context of a <Router>" error

### Pitfall 4: Inline Style Syntax Errors
**What goes wrong:** TypeScript errors on style objects
**Why it happens:** CSS property names in camelCase vs kebab-case, missing units on numeric values
**How to avoid:** Match existing HomePage pattern exactly: backgroundColor not background-color, values as strings with units
**Warning signs:** TypeScript error: "Property does not exist on type CSSProperties"

### Pitfall 5: Session Creation Without Error Handling
**What goes wrong:** Button becomes unresponsive if API call fails
**Why it happens:** Not resetting loading state on error, not displaying error message
**How to avoid:** Wrap fetch in try/catch, set error state, reset loading state in finally block
**Warning signs:** Button stuck in "Creating..." state after network failure

**Example:**
```tsx
const [isCreating, setIsCreating] = useState(false);
const [error, setError] = useState('');

const createSession = async () => {
  setIsCreating(true);
  setError(''); // Clear previous errors
  try {
    const response = await fetch(/* ... */);
    if (!response.ok) throw new Error('Failed to create session');
    const data = await response.json();
    navigate(`/broadcast/${data.sessionId}`, { state: { session: data } });
  } catch (err) {
    setError('Failed to create session. Try again.');
  } finally {
    setIsCreating(false);
  }
};
```

### Session Creation with API Call
```tsx
// Location: HomePage.tsx, add as new function
const [isCreating, setIsCreating] = useState(false);
const [error, setError] = useState('');

const handleCreateBroadcast = async () => {
  const config = getConfig();
  if (!config?.apiUrl) {
    setError('Configuration not loaded');
    return;
  }

  setIsCreating(true);
  setError('');

  try {
    const authToken = localStorage.getItem('token') || '';
    const response = await fetch(`${config.apiUrl}/sessions`, {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${authToken}`,
        'Content-Type': 'application/json',
      },
    });

    if (!response.ok) {
      throw new Error(`API error: ${response.status}`);
    }

    const sessionData = await response.json();
    navigate(`/broadcast/${sessionData.sessionId}`, {
      state: { session: sessionData }
    });
  } catch (err) {
    setError('Failed to create session. Try again.');
  } finally {
    setIsCreating(false);
  }
};
```
