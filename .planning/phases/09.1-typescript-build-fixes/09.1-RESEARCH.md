# Phase 09.1: TypeScript Build Fixes - Research

**Researched:** 2026-03-03
**Domain:** TypeScript compilation, IVS Stage API types, React hook patterns
**Confidence:** HIGH

## Summary

Phase 09.1 addresses concrete TypeScript compilation failures introduced during Phase 8 (RealTime Hangouts) and Phase 7 (Reactions) implementation, plus a test signature mismatch in Phase 5 tests. All errors were identified by running actual builds (`npm run build`) and tests — no guesswork is needed. The exact error messages, line numbers, and file locations are documented below.

There are two categories of bugs: (1) incorrect usage of the `amazon-ivs-web-broadcast` Stage SDK API in `useHangout.ts` and (2) a type boundary mismatch between the frontend and backend `EmojiType` definitions, compounded by `erasableSyntaxOnly: true` forbidding TypeScript `enum` syntax in files consumed by the web build. The backend test failure is a simple signature mismatch — the handler was updated to async-only style but tests still pass 3 arguments.

**Primary recommendation:** Fix each error with the minimum change that satisfies TypeScript — no refactoring, no redesign. Each fix is a targeted 1-10 line change.

## Architecture Patterns

### Issue 1: useHangout.ts — Incorrect Stage API Usage

**File:** `web/src/features/hangout/useHangout.ts`

**Root cause:** The code was written against an assumed API that does not match the installed `amazon-ivs-web-broadcast` package's actual types. Five distinct problems:

#### 1a. StageStrategy type-only import (line 7)

`verbatimModuleSyntax: true` requires type-only imports for types:

```typescript
// WRONG (current)
import { Stage, StageStrategy, SubscribeType } from 'amazon-ivs-web-broadcast';

// CORRECT — StageStrategy is an interface (type), must use import type
import { Stage, SubscribeType, StageEvents } from 'amazon-ivs-web-broadcast';
import type { StageStrategy } from 'amazon-ivs-web-broadcast';
```

Or in a single import using inline `type`:
```typescript
import { Stage, type StageStrategy, SubscribeType, StageEvents } from 'amazon-ivs-web-broadcast';
```

#### 1b. Unused `stage` state variable (line 24)

The `stage` state is set but never used in JSX. Either remove it or keep it for debugging with `_stage` convention. Since the hook returns only `localVideoRef, participants, isJoined, error, toggleMute, toggleCamera`, the `stage` state can be removed entirely.

```typescript
// REMOVE this line:
const [stage, setStage] = useState<Stage | null>(null);
// And remove: setStage(stageInstance);
```

#### 1c. stageStreamsToPublish wrong return type (line 55)

`StageStrategy.stageStreamsToPublish()` must return `LocalStageStream[]`, not `SubscribeType`:

```typescript
// WRONG (current)
stageStreamsToPublish: () => SubscribeType.AUDIO_VIDEO,

// CORRECT — return empty array if publishing from getUserMedia separately
// OR obtain LocalStageStream objects from MediaStreamTrack
stageStreamsToPublish: () => [],
```

**Important context:** The current code gets media via `navigator.mediaDevices.getUserMedia()` and attaches to a video element directly, rather than via `LocalStageStream`. For the compile fix, returning `[]` from `stageStreamsToPublish` is the minimal correct fix. The IVS SDK publishes audio/video when `LocalStageStream` objects are returned. For full runtime correctness (HANG-05, HANG-06), the streams should come from `LocalStageStream` wrapping the `MediaStreamTrack` objects. See Code Examples section for the correct pattern.

#### 1d. Stage.Events does not exist — use imported StageEvents enum (lines 77, 92, 101)

`Stage` class does not have a static `Events` property. The events enum is the separately exported `StageEvents`:

```typescript
// WRONG (current)
stageInstance.on(Stage.Events.STAGE_PARTICIPANT_JOINED, ...)
stageInstance.on(Stage.Events.STAGE_PARTICIPANT_LEFT, ...)
stageInstance.on(Stage.Events.STAGE_PARTICIPANT_STREAMS_CHANGED, ...)

// CORRECT — import StageEvents and use it directly
import { Stage, SubscribeType, StageEvents } from 'amazon-ivs-web-broadcast';
stageInstance.on(StageEvents.STAGE_PARTICIPANT_JOINED, ...)
stageInstance.on(StageEvents.STAGE_PARTICIPANT_LEFT, ...)
stageInstance.on(StageEvents.STAGE_PARTICIPANT_STREAMS_ADDED, ...)  // Note: ADDED not CHANGED
```

**Critical:** The event is `STAGE_PARTICIPANT_STREAMS_ADDED`, not `STAGE_PARTICIPANT_STREAMS_CHANGED`. This is an API name difference, not just a prefix change.

#### 1e. stage.leave() returns void, not Promise (line 148)

From the type definition: `leave: () => void;` (not async):

```typescript
// WRONG (current)
stageInstance.leave().catch(console.error);

// CORRECT — just call leave(), it's synchronous
stageInstance.leave();
```

### Issue 2: useActiveSpeaker.ts — NodeJS.Timeout in Browser Context

**File:** `web/src/features/hangout/useActiveSpeaker.ts`

**Root cause:** `NodeJS.Timeout` is the Node.js type for `setInterval`/`setTimeout` return values. The web tsconfig uses `types: ["vite/client"]` which does not include `@types/node`. In browser context, `setInterval` returns `number`.

```typescript
// WRONG (current, line 29)
let intervalId: NodeJS.Timeout;

// CORRECT — use ReturnType utility or number
let intervalId: ReturnType<typeof setInterval>;
```

`ReturnType<typeof setInterval>` is the canonical cross-environment solution — it works in both Node.js and browser contexts without importing `@types/node`.

### Issue 3: ReplayViewer.tsx — Two Problems

**File:** `web/src/features/replay/ReplayViewer.tsx`

#### 3a. Unused useMemo import (line 5)

```typescript
// WRONG (current)
import { useState, useEffect, useMemo } from 'react';

// CORRECT

```

#### 3b. EmojiType mismatch across package boundary (line 138)

**Root cause:** Two incompatible `EmojiType` definitions exist:

- **Frontend** (`web/src/features/reactions/ReactionPicker.tsx`): `export type EmojiType = 'heart' | 'fire' | 'clap' | 'laugh' | 'surprised';` — string union type
- **Backend** (`backend/src/domain/reaction.ts`): `export enum EmojiType { HEART = 'heart', FIRE = 'fire', ... }` — TypeScript enum

`ReplayViewer.tsx` imports `Reaction` from `backend/src/domain/reaction` (line 17), which uses the backend's `EmojiType` enum. The `handleReaction` function receives the frontend `EmojiType` string union and tries to assign it to `newReaction.emojiType` typed as the backend enum — these are structurally incompatible even though the runtime values are identical strings.

**Additionally**, `erasableSyntaxOnly: true` in `web/tsconfig.app.json` forbids TypeScript `enum` syntax entirely (it cannot be erased — it generates runtime code). The web build fails because it cross-compiles the backend's `reaction.ts` via its project reference/relative import.

**Two-part fix:**

**Part A — Fix backend `reaction.ts` to use const object instead of enum:**

```typescript
// WRONG (current) — enum forbidden by erasableSyntaxOnly
export enum EmojiType {
  HEART = 'heart',
  FIRE = 'fire',
  CLAP = 'clap',
  LAUGH = 'laugh',
  SURPRISED = 'surprised',
}

// CORRECT — const object + type alias (erasable, same runtime values)
export const EmojiType = {
  HEART: 'heart',
  FIRE: 'fire',
  CLAP: 'clap',
  LAUGH: 'laugh',
  SURPRISED: 'surprised',
} as const;
export type EmojiType = typeof EmojiType[keyof typeof EmojiType];
// Result: EmojiType = 'heart' | 'fire' | 'clap' | 'laugh' | 'surprised'
```

**Part B — Remove duplicate frontend EmojiType, import from backend:**

After fixing the backend, the frontend and backend share identical string union values. `ReplayViewer.tsx` can then:

```typescript
// Option A: Change frontend EactionPicker.tsx to re-export from backend
// In ReactionPicker.tsx:
export type { EmojiType } from '../../../../backend/src/domain/reaction';

// Option B: Keep both type aliases (structurally identical, TypeScript will unify them)
// This works because after the enum->const fix, backend EmojiType becomes the
// same string union as the frontend EmojiType, so they ARE assignable.
```

**Recommended:** Option B (keep both type aliases). After fixing backend `enum` to `const`+type, the two `EmojiType` types become structurally identical string unions. TypeScript duck-typing means the type error at line 138 disappears automatically without changing any imports. This is the minimal fix.

Also fix the `ReactionType` enum in `reaction.ts` (line 21 has the same TS1294 error):

```typescript
// WRONG (current)
export enum ReactionType {
  LIVE = 'live',
  REPLAY = 'replay',
}

// CORRECT
export const ReactionType = {
  LIVE: 'live',
  REPLAY: 'replay',
} as const;
export type ReactionType = typeof ReactionType[keyof typeof ReactionType];
```

### Issue 4: stream-started.test.ts — Handler Signature Mismatch

**File:** `backend/src/handlers/__tests__/stream-started.test.ts`

**Root cause:** The test was written when `handler` still accepted the 3-argument Lambda callback signature `(event, context, callback)`. The handler was later updated to async-only `(event): Promise<void>`. The test still passes 3 arguments.

```typescript
// WRONG (current) — passes mockContext and mockCallback as 2nd/3rd args
await expect(handler(event, mockContext, mockCallback)).resolves.not.toThrow();

// CORRECT — handler now takes only event
await expect(handler(event)).resolves.not.toThrow();
```

The `mockContext` and `mockCallback` variables can be removed since they're only used in these two calls.

## Common Pitfalls

### Pitfall 1: Assuming Stage.Events is a static property
**What goes wrong:** Code written assuming `Stage.Events.STAGE_PARTICIPANT_JOINED` pattern (similar to some other SDKs)
**Why it happens:** Convention from other event-driven APIs; IVS exports the events enum separately
**How to avoid:** Import `StageEvents` from the package, use it directly
**Warning signs:** TS2339 "Property 'Events' does not exist on type 'typeof Stage'"

### Pitfall 2: Treating stage.leave() as async
**What goes wrong:** Calling `.catch()` on `void` return value
**Why it happens:** `stage.join()` IS async — developers assume `leave()` mirrors it
**How to avoid:** Check the type definition — `leave: () => void;` is synchronous
**Warning signs:** TS2339 "Property 'catch' does not exist on type 'void'"

### Pitfall 3: stageStreamsToPublish returning SubscribeType
**What goes wrong:** Method returns `SubscribeType.AUDIO_VIDEO` instead of `LocalStageStream[]`
**Why it happens:** Confusion between "what to subscribe to" and "what to publish" — method names differ
**How to avoid:** `stageStreamsToPublish()` → returns `LocalStageStream[]`; `shouldSubscribeToParticipant()` → returns `SubscribeType`
**Warning signs:** TS2322 "Type 'SubscribeType' is not assignable to type 'LocalStageStream[]'"

### Pitfall 4: TypeScript enum with erasableSyntaxOnly
**What goes wrong:** `enum` keyword generates runtime JavaScript (lookup objects), violating the erasable syntax constraint
**Why it happens:** `erasableSyntaxOnly` is a TypeScript 5.5 feature; older code used enum freely
**How to avoid:** Replace `enum Foo { A = 'a' }` with `const Foo = { A: 'a' } as const; type Foo = typeof Foo[keyof typeof Foo]`
**Warning signs:** TS1294 "This syntax is not allowed when 'erasableSyntaxOnly' is enabled"

### Pitfall 5: NodeJS.Timeout in browser TypeScript
**What goes wrong:** `NodeJS.Timeout` type not found because `@types/node` is not in scope for browser builds
**Why it happens:** Node.js and browser `setInterval` return different types historically
**How to avoid:** Use `ReturnType<typeof setInterval>` — works in both environments
**Warning signs:** TS2503 "Cannot find namespace 'NodeJS'"

### Pitfall 6: Lambda callback signature in async handler tests
**What goes wrong:** Test passes `(event, context, callback)` to a handler typed as `(event) => Promise<void>`
**Why it happens:** Handler was refactored from callback style to async/await but tests weren't updated
**How to avoid:** Match test call signature to actual handler signature
**Warning signs:** TS2554 "Expected 1 arguments, but got 3"

## Code Examples

### Correct Stage SDK Import and Event Usage
```typescript
// Source: amazon-ivs-web-broadcast/dist/amazon-ivs-web-broadcast.d.ts
import { Stage, SubscribeType, StageEvents, LocalStageStream } from 'amazon-ivs-web-broadcast';
import type { StageStrategy, StageParticipantInfo, StageStream } from 'amazon-ivs-web-broadcast';

// Strategy with correct stageStreamsToPublish return type
const strategy: StageStrategy = {
  stageStreamsToPublish(): LocalStageStream[] {
    return []; // or wrap MediaStreamTracks in LocalStageStream instances
  },
  shouldPublishParticipant(participant: StageParticipantInfo): boolean {
    return participant.isLocal;
  },
  shouldSubscribeToParticipant(participant: StageParticipantInfo): SubscribeType {
    return SubscribeType.AUDIO_VIDEO;
  },
};

// Correct event listener usage
stageInstance.on(StageEvents.STAGE_PARTICIPANT_JOINED, (participant) => { ... });
stageInstance.on(StageEvents.STAGE_PARTICIPANT_LEFT, (participant) => { ... });
stageInstance.on(StageEvents.STAGE_PARTICIPANT_STREAMS_ADDED, (participant, streams) => { ... });

// Correct leave() — synchronous, no .catch()
stageInstance.leave();
```

### Correct ReturnType Pattern for Timer
```typescript
// Cross-environment timer type
let intervalId: ReturnType<typeof setInterval>;
intervalId = setInterval(() => { ... }, 100);
clearInterval(intervalId);
```

### Correct Enum-to-Const Replacement
```typescript
// Before (forbidden by erasableSyntaxOnly)
export enum EmojiType {
  HEART = 'heart',
  FIRE = 'fire',
}

// After (erasable, structurally same type)
export const EmojiType = {
  HEART: 'heart',
  FIRE: 'fire',
} as const;
export type EmojiType = typeof EmojiType[keyof typeof EmojiType];
// typeof EmojiType = { HEART: 'heart'; FIRE: 'fire' }
// keyof typeof EmojiType = 'HEART' | 'FIRE'
// EmojiType = 'heart' | 'fire'
```

### Correct Async Handler Test Pattern
```typescript
// Handler signature: (event: EventBridgeEvent<...>) => Promise<void>
await expect(handler(event)).resolves.not.toThrow();
```

## Open Questions

1. **Does fixing `stageStreamsToPublish: () => []` break runtime publish?**
   - What we know: The current code gets local media via `getUserMedia` separately and attaches to a video element. The SDK's `stageStreamsToPublish` is supposed to return `LocalStageStream` wrappers around `MediaStreamTrack` objects for the SDK to actually publish audio/video to remote participants.
   - What's unclear: Whether the current Phase 8 implementation ever successfully published local streams to remote participants. Returning `[]` from `stageStreamsToPublish` means the local participant won't publish any tracks.
   - Recommendation: The compile fix (return `[]`) will satisfy the type system. The planner should decide whether Plan 09.1-01 also fixes the runtime publishing by wrapping `MediaStreamTrack` objects in `LocalStageStream` — this is within scope of HANG-05/HANG-06.

2. **Does removing frontend EmojiType enum change any runtime behavior?**
   - What we know: The frontend already uses a string union type alias (not an enum). Only the backend uses enums.
   - What's unclear: Whether any backend Lambda code depends on the enum object shape (e.g., `EmojiType.HEART` as a runtime value vs `'heart'`).
   - Recommendation: After replacing `enum` with `const` + type, usage like `EmojiType.HEART` still works at runtime (accessing `'heart'` from the const object). No behavior change.
